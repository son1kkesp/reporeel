# RepoReel v1 — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Construir la v1 de RepoReel: pegas un repo de GitHub → genera un tráiler vertical 9:16 en MP4 con HyperFrames + copy por IA, con demo pública (galería sembrada + caché + rate-limit) y bucle viral (watermark + OG).

**Architecture:** Hexagonal. `core/` puro y testeable sin red/IA (storyboard + copy fallback). `adapters/` para GitHub, IA (copy), compose (variables HyperFrames), render (Vercel Sandbox → Blob) y caché. `app/` Next.js sirve landing+galería y el flujo async generate/status, con páginas `/r` que emiten OG.

**Tech Stack:** Next.js 16, TypeScript estricto, Tailwind v4, shadcn/ui, HyperFrames + Vercel Sandbox + Vercel Blob, GitHub REST, IA copy vía OpenRouter, Zod, Vitest, Playwright + axe-core, pnpm/Node 24, Vercel.

**Spec:** `docs/superpowers/specs/2026-06-10-reporeel-design.md` — leer antes de empezar (sobre todo §8 contratos, §9 coste, §10 plantilla).

**Reglas globales:** DRY · YAGNI (nada marcado v2 en el spec) · TDD para `core/` · commits atómicos · **verificar con context7 la API actual antes de tocar Next.js 16, HyperFrames, AI SDK/OpenRouter** · **secretos (`OPENROUTER_API_KEY`, `GITHUB_TOKEN`, Blob) solo en `.env.local` / Vercel env, jamás en el repo**.

**Orden de fases:** 0 Scaffold → 1 core → 2 GitHub → 3 plantilla+compose → 4 IA copy → 5 render+caché → 6 app(API+UI+OG) → 7 galería sembrada → 8 deploy.

> **v1.1 — tras revisión del plan:** (1) **Task 5.0 SPIKE** del flujo Vercel Sandbox antes del render; (2) plantilla 3.1 troceada en hitos A-D con contrato de variables primero; (3) `maxDuration = 300` en la route de render (las Functions mueren a 60 s, el render dura ~90 s); (4) creación explícita de `render.mock.ts` + `__fixtures__/sample.mp4`; (5) interfaces `CacheAdapter`/`JobAdapter` antes del código; (6) semáforo = riesgo aceptado acotado; (7) seed-gallery idempotente; (8) `webServer` de Playwright en CI; (9) preflight DNS del subdominio.

## Phase 6 — App (API async + UI + OG)

**Files:** `app/lib/limits.ts`, Test `tests/app/limits.test.ts`

### Task 6.2: API `/api/generate` y `/api/status`

- [ ] **Contrato (spec §8):** `POST /api/generate {owner, repo}` → valida; `jobId`=clave día; si `cache.getTrailer` hit → `{status:'ready', url, poster}`; si rate-limit → 429; si no hay slot → 503; si miss → `job.set('rendering')`, **lanza** el pipeline (github→storyboard→script(+fallback)→compose→render→cache→job.set('ready')), devuelve `{jobId, status:'rendering'}`. `GET /api/status?jobId=` → lee `job.get`.
- [ ] **Tests (adapters mockeados, sin IA/render real):** (a) cache hit → ready; (b) miss → rendering + jobId; (c) rate-limit → 429; (d) sin slot → 503; (e) render mock que falla → `job` queda `error`, status lo refleja. ciclo+commit.
- [ ] **`maxDuration`:** la route que ejecuta el render declara `export const maxDuration = 300` (Fluid Functions, plan Pro) — una Function normal muere a los 60 s y el render dura ~90 s. Verificar API Next.js 16 con context7 (route handlers, `cookies()` async, `maxDuration`).
